# Classnect Developer Onboarding Manual

## 1. System Overview
Classnect is a global **EdTech SaaS** dedicated to making complex mathematics intuitive through high-fidelity visual learning. Our growth engine is built on a **Product-Led Growth (PLG)** model, utilizing a **Freemium** approach to convert guest learners into active community members.

### Core Value Proposition
- **Global Multi-Track Curriculum**: A unified engine capable of serving diverse educational systems (French Baccalaureate, AP Calculus, etc.).
- **Visual Intuition**: Replacing static text with interactive, animated mathematical concepts.
- **Global Accessibility**: Sub-millisecond latency via edge-computing infrastructure.

---

## 2. Tech Stack
Our architecture is "Edge-First," designed for maximum speed and global distribution.

- **Frontend**: 
  - [Vite](https://vitejs.dev/) + [React](https://react.dev/) (TypeScript)
  - [Tailwind CSS](https://tailwindcss.com/) for Atomic Design.
  - [react-i18next](https://react.i18next.com/) for UI Shell localization.
- **Backend**: 
  - [Cloudflare Workers](https://workers.cloudflare.com/) for Edge execution.
  - [Hono](https://hono.dev/) as the ultrafast, type-safe API framework.
- **Database**: 
  - [Cloudflare D1](https://developers.cloudflare.com/d1/) (Edge-native SQLite).
  - [Drizzle ORM](https://orm.drizzle.team/) for schema management and type-safe queries.
- **Authentication**: 
  - [Firebase Auth](https://firebase.google.com/docs/auth) handles identity; UIDs are synced to the local D1 `users` table.

## 4. Database Schema
We utilize Cloudflare D1 for relational storage. Our schema is defined in `src/worker/db/schema.ts`.

### `users`
Syncs with Firebase Auth to store learner profile and track selection.
- `id` (PK): Firebase UID.
- `email`: Primary contact.
- `displayName`: Learner's public name.
- `photoUrl`: R2 or Google-hosted avatar.
- `plan`: 'FREE' or 'PREMIUM'.
- `currentTrackId`: The active curriculum track (e.g., `ma-2bac-math`).

### `progress`
Atomic tracking of lesson completion.
- `userId` (FK): Links to `users.id`.
- `unitId`: String identifier from curriculum JSON.
- `isCompleted`: 0 (False) or 1 (True).
- `score`: Total points earned in this unit.

### `feedback`
Direct user feedback loop.
- `userId` (FK): Submitting user.
- `type`: 'idea' | 'bug' | 'complaint'.
- `message`: The content of the feedback.
- `status`: 'pending' | 'reviewed' | 'resolved'.

### `error_logs`
Automated telemetry for platform stability.
- `message`: Error description.
- `stack`: Serialized stack trace.
- `signature`: Unique MD5/String hash for deduplication.
- `occurrences`: Incrementing counter for frequency analysis.

## 5. Core Architectural Concepts

### Hybrid i18n Strategy (Shell vs. Content)
We maintain a strict separation between the **Application Shell** and the **Curriculum Content**:
1. **UI Shell**: Navigation, buttons, and settings are handled by `react-i18next`. It supports dynamic RTL/LTR switching for Arabic support.
2. **Curriculum Content**: Lesson text and math problems are served **exactly** as they appear in the track's JSON file. 
   - *Rationale*: A student studying for the "AP Calculus" (English) track should see mathematical definitions in English, even if their "Profile" button is translated to French.

### Dynamic Curriculum Loading
The `curriculumLoader.ts` utility facilitates dynamic fetching. Instead of bundling all educative data, the app detects the user's `currentTrackId` and asynchronously loads only the relevant JSON from `src/react-app/data/curriculum/`. This ensures the initial bundle size remains small as we add dozens of new tracks.

### Signature-Based Error Deduplication
To avoid database bloat from repetitive client-side crashes, our **Global Error Boundary** implements a signature-based mechanism:
- When an error occurs, it generates a "signature" (a combination of the message and the first few lines of the stack trace).
- It checks a local `Set` to see if this error was already logged in the current session.
- If unique, it sends the error to the backend, where the D1 `error_logs` table increments the `occurrences` count rather than creating a new row if the signature already exists.
